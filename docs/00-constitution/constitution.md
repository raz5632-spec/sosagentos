# SalesOS Constitution

SalesOS is **not** a CRM, not an LMS, not a chatbot, and not a generic AI wrapper.
It is a **digital operating company for S.O.S.** — centralizing knowledge, decisions, student
operations, content production, research, competitor intelligence, analytics, automation,
and communications under one system. The product is **decision-centric**, not task-centric:
it surfaces what should happen and why, then executes, queues for approval, or recommends —
depending on governance level.

## Non-negotiables

| Area | Decision |
|---|---|
| Product identity | SalesOS is a multi-agent AI Operating System for S.O.S. |
| Architectural stance | A company of specialized agents, not one omnipotent chat agent |
| Control model | AI suggests and prepares; humans retain final authority over material business actions |
| Business rule | Every feature must save time, improve quality, or improve efficiency; and be automatable and scalable |
| Information model | One Source of Truth; no duplicate canonical records |
| Knowledge model | capture → validation → categorization → relationship mapping → approval → production → versioning |
| Safety | AI never guesses when confidence is low; it asks, searches, or says it does not know |
| Reversibility | Every meaningful action must be reversible or compensatable |
| Auditability | Everything is logged: prompts, tool calls, outputs, approvals, rejections, external API actions |
| Memory design | Each agent has isolated working memory; promotion to central memory is controlled |
| Governance | Approval levels (L0–L4) determine execute / notify / ask / recommend-only |
| Integrations | Only official / compliant APIs and approved flows; no scraping that violates platform terms |
| Product scope | Build platform core for S.O.S. first, but architect for multi-tenant future packages |
| Explainability | The system must explain why it recommended or executed an action |
| Execution model | Claude Code acts as CTO + TPM + senior engineer, task-by-task against the Execution Book |
| Delivery discipline | No coding without architecture, acceptance criteria, tests, docs, and rollback considerations |
| Future-proofing | Vendor-agnostic model router; no hard dependency on a single LLM provider |
